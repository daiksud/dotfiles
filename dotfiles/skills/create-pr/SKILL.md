# create-pr

In this `SKILL.md`, "Pull Request" is abbreviated as "PR".

## Overview

This skill allows an AI agent to create a draft PR on behalf of the developer.
It understands the content of staged or committed files, composes appropriate
commit messages and PR descriptions, and creates the PR.

## When to Use

- When asked to "create a PR"
- When invoked via `/create-pr` in GitHub Copilot CLI

## How to Use

### Step 1: Deeply understand the changes

- Run `git diff --staged` or `git diff origin/main..` to review the full diff
- Read each file's diff to understand **what** changed and **why**
- Consider any context or rationale passed when the skill was invoked
- For new files, read the entire file to grasp its purpose and role
- If changes span multiple logical units, consider splitting into multiple commits

### Step 2: Create commit message(s) and commit

- Run `git diff --staged --quiet`; only commit if exit code is 1 (i.e., staged files exist)
- Follow [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) format
- Write commit messages in English
- Run `git log --no-merges --oneline -100` to review recent commits and match repository conventions
- **The commit subject must concretely describe the change**
  - Bad: `docs: apply staged changes` (unclear what changed)
  - Bad: `feat: update files` (too vague)
  - Good: `feat: allow provided config object to extend other configs`
  - Good: `fix(github): pin terraform-provider-github version to 6.6.0`
- Set scope to indicate the area of impact (directory, action name, etc.)
- Use imperative mood (add, fix, update) so the intent is clear
- Add a body with supplemental explanation when needed
- Splitting into multiple commits is fine when it aids review or logical separation

### Step 3: Push to a feature branch

- If currently on main, create and checkout a feature branch
- Branch name should reflect the change (e.g., `feat/add-create-pr-skill`, `fix/pin-terraform-provider-github-660`)
- Push with `git push --set-upstream origin <branch>`

### Step 4: Write the PR description and create a draft PR

- If `.github/pull_request_template.md` exists, follow its structure. Add the following line at the top (after any leading HTML comment): `_This PR was generated by GitHub Copilot CLI using /create-pr skill._`
- If no template exists, use that line as the first line of the description
- The description should include:
  - **Purpose**: What this PR achieves (include "why", not just "what")
  - **Background**: Context on why this change is needed
  - **Related Issues**: Link any related issues
- If `.github/CONTRIBUTING.md` exists, follow its guidelines and avoid these anti-patterns:
  - Listing "files changed" verbatim (don't state what's obvious from the diff)
  - High-context explanations without providing reasoning
- Create the PR with `gh pr create --draft`
  - Set labels appropriate to the PR content (e.g., `feature`, `bug`, `documentation`)

### Step 5: Assign the requesting user

- Set the user who invoked this skill as the PR Assignee
- Use `gh pr edit <PR_NUMBER> --add-assignee <username>`

## Output

Display the URL of the created draft PR.

## Tips for developers

Register a shell alias (e.g., in `~/.bashrc`) for convenient invocation from a terminal with [GitHub Copilot CLI](https://github.com/github/copilot-cli) installed:

Setup:

```bash
alias create-pr='f() { copilot --model ${COPILOT_MODEL:-claude-sonnet-4.6} -p "/create-pr skill $*"; }; f'
```

Simplest usage:

```bash
create-pr
```

With additional context:

```bash
create-pr The reason for the change is XX, add #1476 as a related task
```
